# Inception 最強ガイドブック：提出までこれ一冊

このガイドブックは、42の課題「Inception」を単なる作業（コピペ）ではなく、**「システム設計の理解」**を伴って完遂し、ピア評価で自信を持って説明できるように構成されています。

## 0. プロジェクトの全体像（三層アーキテクチャ）

Inceptionは、現代のWebインフラの標準である**三層アーキテクチャ**をDockerで再現する課題です。

1.  **プレゼンテーション層 (NGINX)**: TLS 1.2/1.3で通信を保護し、リクエストを後段へ受け渡す。
2.  **アプリケーション層 (WordPress + PHP-FPM)**: WP-CLIで自動インストールされ、ロジックを実行する。
3.  **データ層 (MariaDB)**: データベース専用。外部ネットワークからは隔離されている。

---

## 1. 基礎知識と必須ルール

設計を始める前に、不合格（Fail）に直結するルールを再確認してください。

- **OS**: Debian (penultimate stable: trixie or bookworm) または Alpine。
- **NGINX**: ポート443以外からアクセスさせてはいけない。
- **Docker Network**: `network: host` は禁止。コンテナ間は名前解決 (`mariadb`, `wordpress` など) で通信する。
- **Volumes**: バインドマウント禁止。ホストの `/home/login/data` に保存される「ネームドボリューム」を使用する。
- **PID 1**: コンテナ内のメインプロセスが PID 1 であること（詳細は 4章参照）。
- **ハック禁止**: `tail -f`, `sleep infinity` など、無限ループでコンテナを維持させてはいけない。

## 2. インフラ設定の重要ポイント

### 2.1 ドメイン解決 (`/etc/hosts`)
VM内とMac/Windows側の両方で `/etc/hosts` に以下を追記します。
```text
127.0.0.1   <login>.42.fr
```

### 2.2 秘密情報の管理（.env vs. Docker Secrets）

課題要件では環境変数の使用が必須ですが、機密情報については **Docker Secrets** の使用が強く推奨されています。

| 特徴 | .env (環境変数) | Docker Secrets |
| :--- | :--- | :--- |
| **データの形** | `KEY=VALUE` 形式の変数 | **ファイル** として存在 |
| **コンテナ内の場所** | プロセスの環境メモリ | `/run/secrets/<名前>` というファイル |
| **可視性** | `docker inspect` で丸見え | `inspect` では中身が見えない |
| **保存先** | コンテナ設定（ディスク）に残る | **メモリ上(tmpfs)** にのみ展開される |

**なぜ Secrets が最強なのか？**
- **漏洩リスクの低減**: アプリのログや `docker inspect` でパスワードが露出するのを防げます。
- **物理的な保護**: コンテナ内の `/run/secrets/` は RAM ディスク（tmpfs）上にあり、コンテナ停止時に跡形もなく消去されます。
- **権限管理**: ファイルとして渡されるため、Linux のファイル権限を使って「mysql ユーザーのみが読み取れる」といった制御が可能です。

評価（Defense）では、「なぜ Secrets を使っているのか？」という問いに対し、**「メモリ上（tmpfs）に展開されるため永続ストレージに跡を残さない点」**や**「inspect コマンドによる露出を防げる点」**を回答できるようにしておきましょう。

### 2.3 コンテナのライフサイクルとスクリプトの実行

「コンテナが立ち上がる時」とは、具体的に以下のコマンドが実行されたタイミングを指します。

- **`docker run`**: イメージから新しいコンテナを作成し、起動する。
- **`docker start`**: 停止している既存のコンテナを再び起動する。
- **`docker-compose up`**: 設定に従ってコンテナを作成・起動する。

**なぜ再起動を考慮する必要があるのか？**
コンテナは「使い捨て」が基本ですが、データベースなどのデータは永続化（ボリューム）されます。
`init.sh` や `setup.sh` などのエントリーポイントスクリプトが、**「コンテナが起動するたびに実行される」** という特性を理解しておくことが重要です。

- **初回起動時**: データベースの作成や WordPress のインストールを行う。
- **2回目以降（再起動）**: すでにデータが存在するため、初期化処理をスキップし、メインプロセス（MariaDB や PHP-FPM）の起動だけを行う。

この「初回か否かの判定」をスクリプト内に組み込むことで、データの二重作成や上書きといったバグを防ぐことができます。

### 2.4 コンテナ間通信と Docker Network

コンテナ間通信とは、独立して動いているコンテナ同士がネットワークを通じてデータをやり取りすることです。

**Docker Network の役割**
- **仮想ネットワークの構築**: `docker-compose.yml` でネットワークを定義することで、コンテナ専用の「仮想的な部室」が作られます。
- **名前解決 (DNS)**: コンテナ同士は IP アドレスではなく、**サービス名（mariadb, wordpress など）** でお互いを呼び出すことができます。
- **セキュリティ**: 外部（インターネット）には公開せず、コンテナ間だけで通信を完結させることができます（例：MariaDB は WordPress とだけ話し、外の世界とは直接話さない）。

評価（Defense）では、「なぜ WordPress は MariaDB の IP アドレスを知らなくても接続できるのですか？」という質問に対し、**「Docker の内蔵 DNS がサービス名を IP アドレスに解決してくれるからです」** と答えられるようにしておきましょう。
### 2.5 ポート番号とポートマッピングの概念

ポート番号とは、ネットワーク経由で届いたデータを、コンピューター内の「どのプログラム」に渡すかを決めるための識別子です。

**送信先ポート (Destination Port) と送信元ポート (Source Port)**
通信は常に一対のポート番号で管理されます。

- **送信先ポート (宛先)**: サーバー側が待ち受けている固定の番号（例：SSH なら 22, HTTPS なら 443）。
- **送信元ポート (発信元)**: クライアント側が一時的に使用するランダムな番号（**エフェメラルポート**）。通常 49152〜65535 の範囲から割り当てられます。

**通信の識別 (4-tuple) と双方向性**
OS は以下の 4 つの要素の組み合わせ（4-tuple）で一つの通信セッションを特定します。
1. 送信元 IP / 2. 送信元ポート / 3. 送信先 IP / 4. 送信先ポート

- **双方向通信の成立**: 一度接続が確立されると、同じポートのペア（例：クライアント 54321 ↔ サーバー 22）を使い続けて双方向のデータ転送を行います。返信時は送信元と送信先を入れ替えるだけで、新しいポート番号を消費することはありません。
- **固定 vs ランダム**: 
    - サーバー側（受信側）は、サービス提供場所を固定する必要があるため、特定の番号（ウェルノウルポート）で「待ち受け（Listen）」ます。
    - クライアント側（送信側）は、重複さえしなければ番号は何でも良いため、OS が動的に「エフェメラルポート」を割り当てます。

**Docker におけるポートマッピング (`ports:` / `-p`)**
Docker Compose 等で指定する `2222:22` という設定は、**「ホストマシンの窓口」と「コンテナ内の窓口」を直通のパイプで繋ぐ** ことを意味します。

- **Host Port (左側 / 例: 2222)**: 外部（Mac のターミナル等）からアクセスする際の「入り口」。
- **Container Port (右側 / 例: 22)**: コンテナの中でプログラム（SSH サーバー等）が実際に「待ち受けている」番号。

**注意点**
- **宛先ポートの固定**: `-p 2222:22` と設定した場合、外部からこのコンテナにアクセスする際の **宛先ポート（Destination Port）** が 2222 番になります。
- **送信元ポートへの影響なし**: 通信を発信する側（クライアント）のポートは、この設定とは無関係に OS がランダムに割り当てます。

**Inception での注意点**
- **エントリポイントの原則**: 外部に公開する（`ports:` に書く）のは、唯一のエントリポイントである **NGINX (443)** だけにするのがセキュアな設計です。
- **内部通信**: WordPress や MariaDB は、Docker Network 内だけで通信するため、`ports:` で外部に公開する必要はありません。


### 2.6 プロセス間通信 (IPC) vs ネットワーク通信 (TCP/IP)

同じコンピューター内での通信には「ソケット (IPC)」と「ポート (TCP/IP)」の 2 つの選択肢がありますが、その内部処理には大きな差があります。

| 特徴 | プロセス間通信 (UNIX ソケット) | ネットワーク通信 (TCP ポート) |
| :--- | :--- | :--- |
| **OS 内部のパス** | カーネルによるメモリの直接コピー | プロトコルスタック（TCP/IP 層）を通過 |
| **処理負荷** | 最小（ヘッダ付与や計算が不要） | 高（チェックサム計算やパケット分割が必要） |
| **信頼性** | メモリ上なので 100% 保証（再送制御不要） | TCP ロジックによる確認と再送制御が必要 |
| **セキュリティ** | ファイル権限によるアクセス制御 | IP/ポート制限（Firewall 等）による制御 |

**なぜ Inception で使い分けるのか？**
- **コンテナ間 (WP ↔ MariaDB)**: 物理的に異なるコンテナ（家）のため、ネットワークスタックを通る **TCP ポート (3306)** を使うしかありません。
- **コンテナ内 (管理者 ↔ MariaDB)**: 同じコンテナ（部屋）の中なので、オーバーヘッドのない爆速な **UNIX ソケット** を利用して効率的に操作を行います。

### 2.7 環境変数のバトンリレー（変数の流れ）

`${MYSQL_ROOT_PASSWORD}` などの変数が、ホストマシンからコンテナ内のスクリプトに届くまでの流れを理解することは、トラブルシューティングにおいて不可欠です。

1. **`.env` (ホスト)**: 実際の値を定義する場所。
2. **`docker-compose.yml`**: ホストの値を読み込み、コンテナの `environment:` セクションへ受け渡す。
3. **Container OS**: コンテナ起動時、OS の環境変数として値がセットされる。
4. **`init.sh` 等のスクリプト**: シェルが変数 `${...}` を実際の値に展開して実行する。

**未設定時の挙動とリスク**
変数が `.env` 等で定義されていない場合、シェルはそれを **「空文字列」** として処理します。
- **現象**: スクリプト自体はエラーにならずに終了し、コンテナが起動することがあります。
- **結果**: 「パスワードなし」のユーザーが作成されるなど、セキュリティ要件を満たさない状態（不合格対象）になります。

必ず `docker exec <container> env` コマンドを使用して、意図した変数がコンテナ内に届いているか確認する癖をつけましょう。

### 2.8 Docker コンテナの権限と root ユーザー

Docker コンテナはデフォルトで **root ユーザー** として起動します。そのため、`init.sh` や `setup.sh` などのスクリプトはシステム全体を操作できる強力な権限を持っています。

**なぜ最初は root が必要なのか？**
- **ディレクトリの作成と権限設定**: `/run/mysqld` の作成や `chown` による所有者変更など、OS のシステムに関わる初期設定を行うためです。

**特権の放棄 (Privilege Dropping)**
セキュリティのベストプラクティスとして、メインのサービス（MariaDB 等）は root 以外の一般ユーザーで動かすべきです。
- **実装例**: `exec mysqld --user=mysql`
- **仕組み**: 初期設定は root で行い、最後にサービスを起動する瞬間に `mysql` ユーザーに権限を移譲します。これにより、万が一サービスが攻撃されても、被害をコンテナ内の特定のユーザー範囲に限定することができます。

評価（Defense）では、「なぜスクリプト内で chown をしているのですか？」や「MariaDB はどのユーザーで動いていますか？」という質問に対し、**「初期設定を root で行い、サービス実行時はセキュリティのために mysql ユーザーに権限を下げているからです」** と回答できるようにしましょう。

### 2.9 PID 1 (Init プロセス) とシグナル管理

PID 1 とは、Linux システムおよび Docker コンテナ内で最初に起動される **「すべてのプロセスの祖先」** です。

**PID 1 の特別な責任**
- **コンテナの生命維持**: PID 1 のプロセスが終了すると、Docker コンテナ全体も終了します。
- **ゾンビプロセスの回収**: 親プロセスがいなくなった不要なプロセスを掃除する役割を担います。
- **シグナルの受信**: `docker stop` 時に送られる `SIGTERM`（終了信号）を最初に受け取るのは PID 1 です。

**Inception での課題と解決 (exec コマンド)**
もし `init.sh` が PID 1 のままだと、終了信号を適切に処理できず、データ破損や終了時のタイムアウト（10秒待機）が発生します。

- **解決策**: `exec mysqld` のように `exec` を使用することで、シェルスクリプトのプロセスを MariaDB プロセスに **「置き換え」** ます。
- **効果**: MariaDB が PID 1 になることで、終了信号を正しく解釈し、データを確実にディスクへ書き込んでから安全に終了（Graceful Shutdown）できるようになります。

評価（Defense）で「なぜ exec を使っているのですか？」と聞かれたら、**「MariaDB を PID 1 に据えることで、OS からの終了信号を正しく受け取り、安全なシャットダウンを実現するためです」** と回答しましょう。

## 4. エントリポイントの「魔法」: PID 1 と `exec "$@"`

コンテナが正しく終了せず、`docker stop` で10秒待たされるのは、シェルスクリプトが PID 1 になっているからです。

**最強のエントリポイント・テンプレート:**
```bash
#!/bin/bash
# 1. 必要な初期設定（DB作成やWPインストールなど）を書く
echo "Configuring service..."

# 2. 最後にメインプロセスに PID 1 を譲る
exec "$@"
```
`exec "$@"` を使うことで、シェルスクリプトのプロセスが消滅し、Dockerfileの `CMD` に書いたプログラム（`php-fpm` や `nginx`）が PID 1 に置き換わります。

## 5. 自動化 (Makefile)

Makefileは、評価者が最初に触るインターフェースです。

- `make`: ディレクトリ作成 ➔ `docker-compose up --build -d`
- `make clean`: コンテナ停止とネットワーク削除
- `make fclean`: イメージ、ボリューム、データディレクトリも含めた完全削除

## 6. トラブルシューティング

| エラー | 原因の切り分け |
| :--- | :--- |
| **502 Bad Gateway** | NGINX ➔ WordPress 間の接続失敗。PHP-FPMが動いているか、ポート9000が開いているか確認。 |
| **Database Connection Error** | WordPress ➔ MariaDB 間の接続失敗。パスワードの不一致、またはDB起動完了前にWPがアクセスした。 |
| **File Not Found (NGINX)** | 静的ファイル用ボリュームが NGINX と WordPress の両方にマウントされているか確認。 |
