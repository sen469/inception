# 第0章: 仮想化技術の変遷とDockerのアーキテクチャ

本章では、Dockerという技術がなぜ生まれ、どのような課題を解決してきたのか、そしてその裏側で動いているOSレベルの仮想化技術の仕組みについて深く掘り下げていきます。「単にコマンドを叩けば動く魔法の箱」ではなく、システムの内部で何が起きているのかを理解することは、トラブルシューティングやインフラ設計において不可欠なスキルとなります。

## 0.1 仮想化の歴史とコンテナの誕生

システム管理の歴史は、ハードウェアリソースをいかに効率的かつ安全に分割・活用するかという課題との戦いでした。

### 物理サーバー時代
かつては、1つのサービス（例えばWebサーバーやデータベース）を稼働させるために、1台の物理的なサーバーハードウェアを割り当てていました。このアプローチには以下の致命的な欠点がありました。
*   **リソースの無駄遣い:** CPUやメモリの使用率が10%程度でも、残りの90%は遊休状態となる。
*   **スケーラビリティの欠如:** トラフィック増大時に、物理的な機器の調達からキッティングまで数週間を要する。
*   **依存関係の衝突:** 1台のサーバーに複数のアプリケーションを同居させると、ライブラリのバージョン（例: Python 2系と3系）の競合が発生する。

### ハイパーバイザ型仮想化（VM）の台頭
これらの課題を解決するために登場したのが VMware, Hyper-V, VirtualBox などの「ハイパーバイザ型仮想化」です。
物理ハードウェアの上にハイパーバイザと呼ばれる仮想化レイヤーを敷き、その上で複数の「ゲストOS」を稼働させます。これによりリソースの分割と環境の隔離に成功しました。
しかし、ゲストOSごとに独立したカーネル（OSの中核）を起動する必要があるため、**起動に数分かかり、1つのVMあたり数GBのメモリを消費する**という「重量級」の問題が残りました。

### コンテナ型仮想化の革新
ここで登場したのがコンテナ技術です。コンテナは、ハイパーバイザのように完全なOSをエミュレートするのではなく、**ホストOSのカーネルを複数のコンテナで共有**しつつ、プロセスやネットワーク、ファイルシステムを論理的に隔離するアプローチをとります。
これにより、OSの起動プロセスを省略できるため**数ミリ秒での起動**が可能となり、メモリ消費もプロセスが実際に使う分（数MB〜）に抑えられるという圧倒的な軽量化を実現しました。

| 比較項目 | ハイパーバイザ型仮想化 (VM) | コンテナ型仮想化 (Docker等) |
| :--- | :--- | :--- |
| **アーキテクチャ** | ゲストOSごとにカーネルを持つ | ホストOSのカーネルを共有 |
| **起動時間** | 分単位 | ミリ秒〜秒単位 |
| **リソース消費** | 大（ゲストOSのオーバーヘッドを含む） | 小（アプリケーションの実行分のみ） |
| 隔離レベル | ハードウェアレベルの完全な分離 | プロセスレベルの論理的な分離 |

*Inceptionの課題において、あなたが作成したVMの中でさらにDockerを動かす理由は、この「重量級のハードウェア仮想化（VM）」と「軽量なアプリケーション隔離（コンテナ）」の二つの層を実践的に理解させるためです。*

## 0.2 Dockerの「三位一体」ワークフロー

Dockerを使いこなす上で絶対に理解しなければならないのが、**「Dockerfile」「イメージ」「コンテナ」**の3つの関係性です。Dockerの作業は常にこの一方通行のフローで進みます。

1.  **Dockerfile (手順書・ソースコード)**
    *   人間が記述するテキストファイルです。「どのOSをベースにするか」「どのソフトウェアをインストールするか」といった**コンテナの作り方の手順**をコードとして定義します。
2.  **Image (イメージ・設計図の実体)**
    *   Dockerfileをビルド（`docker build`）することで生成される、**読み取り専用**の静的なパッケージです。プログラムの実行に必要なすべてのファイルが固まった「スナップショット」であり、一度作られたら中身が変わることはありません（たい焼きの「金型」に相当します）。
3.  **Container (コンテナ・動く実体)**
    *   イメージを実行（`docker run`）することで生成される、**実際にプロセスが稼働している隔離空間**です。1つのイメージ（金型）から、複数のコンテナ（たい焼き）を同時にいくつでも作り出して動かすことができます。

つまり、私たちがサーバー上で実際に動かしているのは「イメージ」ではなく、イメージから作られた「コンテナ」です。

## 0.3 Dockerを支えるLinuxカーネル技術

Dockerは全く新しい魔法の技術を発明したわけではありません。Linuxカーネルに昔から備わっていた機能を、人間が使いやすい形（APIやCLI）にパッケージングしたものです。コアとなる3つのカーネル技術を理解しましょう。

### 1. Namespaces (名前空間)
Namespacesは、特定のプロセスに対して「自分専用のシステムを占有している」ように見せかける技術です。
*   **PID Namespace:** プロセスIDを隔離します。コンテナ内に入って `ps` コマンドを打つと、コンテナ内で実行されているプロセスだけが見え、最初のプロセスには PID 1 が割り当てられます。
*   **NET Namespace:** ネットワークインターフェースやIPアドレス、ルーティングテーブルを隔離します。これにより、各コンテナが独自のIPアドレス（例: `172.17.0.2`）を持つことができます。
*   **MNT Namespace:** マウントポイント（ファイルシステム）を隔離します。

### 2. cgroups (Control Groups)
Namespacesが「見える範囲」を制限するものなら、cgroupsは「使えるリソースの量」を制限・監視する技術です。
特定のコンテナが暴走してCPUを100%占有したり、メモリを使い果たしてホストOS全体をクラッシュ（OOM: Out Of Memory）させるのを防ぐため、「このコンテナはメモリ最大512MB、CPUは0.5コアまで」といった物理リソースの制約をかけます。

### 3. UnionFS / OverlayFS (ユニオン・ファイルシステム)
Dockerイメージの超高速なビルドや軽量な保存を実現しているのがこのストレージ技術です。
複数のディレクトリ（レイヤー）を、透過的に重ね合わせて（オーバーレイして）1つのファイルシステムとして見せることができます。
コンテナを起動すると、読み取り専用の「イメージレイヤー」群の上に、書き込み可能な「コンテナレイヤー」が薄く乗ります。コンテナ内でファイルを変更しても、それは一番上のレイヤーにのみ記録され、元のイメージは一切汚染されません（Copy-on-Write方式）。

## 0.3 Docker Engineのアーキテクチャ

私たちが普段 `docker` コマンドを叩くとき、裏側では複数のコンポーネントが連携して動いています。

1.  **Docker CLI (クライアント):** ユーザーがコマンドを入力するインターフェース（`docker run ...`）。
2.  **Docker Daemon (dockerd):** ホスト上で常駐し、クライアントからのREST APIリクエストを受け付け、イメージの管理やネットワークの構築を行う司令塔。
3.  **containerd:** 実際にコンテナのライフサイクル（起動、停止）を管理する業界標準のコンテナランタイム。
4.  **runc:** Linuxのカーネル機能（Namespaces, cgroups）を直接操作し、低レベルなコンテナ生成を行うモジュール。

このように、Dockerは単一の巨大なプログラムではなく、役割ごとに疎結合に設計された堅牢なシステムとして成り立っています。

# 第1章: Docker操作の実践とコンテナ・ライフサイクル

第0章で学んだ理論を基に、本章では実際にDockerエンジンを操作し、コンテナのライフサイクル（作成、起動、停止、破棄）を制御する方法を学びます。単にコマンドを暗記するのではなく、「そのコマンドを打った時、裏でデーモンが何を行っているのか」を意識しながら進めてください。

## 1.1 Dockerのアーキテクチャへのアクセス

まずは、Dockerデーモンが正常に稼働しており、クライアント（CLI）が通信できる状態かを確認します。

```bash
docker info
```
このコマンドは、Dockerエンジンの現在の状態（稼働中のコンテナ数、ストレージドライバとして `overlay2` が使われているか、カーネルのバージョンなど）を出力します。もしエラーが出る場合は、デーモン（`dockerd`）が起動していないか、ユーザーにソケット（`/var/run/docker.sock`）へのアクセス権限がありません。

## 1.2 `docker run` の深層

「コンテナを起動する」際にもっとも頻繁に使われるのが `docker run` コマンドですが、これは実は複数の処理を一つにまとめたマクロ的なコマンドです。

```bash
docker run -d -p 8080:80 --name my-web nginx:latest
```

この一行を実行した際、Dockerの内部では以下の3つのステップが連続して行われます。

1.  **Pull (イメージの取得):**
    ホストのローカルキャッシュ内に `nginx:latest` のイメージが存在するか確認します。無ければ、デフォルトのレジストリ（Docker Hub）からダウンロード（Pull）します。
2.  **Create (コンテナの生成):**
    イメージを基に、新しい読み書き可能なコンテナレイヤー（Copy-on-Write）を作成し、名前空間（Namespaces）やcgroupsの設定を割り当てて隔離された環境を準備します。この段階ではまだプロセスは起動していません。
3.  **Start (プロセスの起動):**
    Dockerfileに定義されたデフォルトのコマンド（NGINXの場合は `nginx -g "daemon off;"`）を実行し、コンテナを起動します。

### オプションの解説
*   `-d` (Detached mode): コンテナをバックグラウンドで実行し、標準出力をターミナルから切り離します。WebサーバーやDBなどのデーモンプロセスを動かす際に必須です。
*   `-p 8080:80` (Port mapping): ホストOSの8080番ポートへのトラフィックを、コンテナの80番ポートへ転送（NAT）します。内部的にはLinuxの `iptables` が自動的に書き換えられ、ルーティングが行われています。
*   `--name my-web`: ランダムな識別子（ID）の代わりに、人間が判別可能な名前を付けます。

## 1.3 名前空間への潜入（`docker exec`）

コンテナは独立した環境ですが、トラブルシューティングやデバッグのために「実行中のコンテナの中に入って操作したい」という場面が多々あります。

```bash
docker exec -it my-web bash
```

*   `exec`: 実行中のコンテナ内で、新たなプロセスを追加で起動します。
*   `-it`: `-i` (Interactive: 標準入力を開いたままにする) と `-t` (TTY: 擬似端末を割り当てる) の組み合わせです。これにより、コンテナ内で起動したシェル（この場合は `bash`）と対話的に通信できるようになります。

このコマンドは「VMにSSHログインする」ことと似ているように見えますが、技術的には全く異なります。実際には、ホストOSから新しい `bash` プロセスを起動し、そのプロセスを対象のコンテナと同じNamespace（PID, NET, MNT）に参加させているだけです。

## 1.4 コンテナの停止とライフサイクル管理

コンテナの役割が終わったら、安全に停止させる必要があります。

```bash
docker stop my-web
```

`docker stop` を実行すると、Dockerデーモンはコンテナ内のメインプロセス（PID 1）に対して **`SIGTERM`** というシグナル（「終了してください」という通知）を送信します。プロセスが適切に設計されていれば、これを受け取ってメモリの解放やセッションのクローズなどを行い、安全に終了（Graceful Shutdown）します。
10秒待ってもプロセスが終了しない場合、Dockerは **`SIGKILL`**（強制終了）を送信し、プロセスを強制的にキルします。

停止したコンテナはディスク上に存在し続けます（`docker ps -a` で確認可能）。完全に破棄するには以下のコマンドを使用します。

```bash
docker rm my-web
```

## 1.5 一括クリーンアップ

開発を続けていると、停止したコンテナや不要になったイメージ（dangling images）がストレージを圧迫します。これらを一括で安全に削除するのが `prune` コマンドです。

```bash
docker system prune -a
```
このコマンドは、停止しているすべてのコンテナ、使われていないネットワーク、およびアタッチされていないすべてのイメージを削除します。Inceptionの課題で「環境を完全にリセットしてゼロからやり直したい」場合に非常に強力なツールとなります。

## 1.6 まとめ
本章では、イメージからコンテナを生成し、操作し、破棄するという一連のライフサイクルを学びました。次章では、コンテナの元となる「イメージ」がどのように流通し、どのような基準でベースOSを選択すべきかについて解説します。
# 第2章: イメージの流通とベースOSの選択

Dockerの真の強力さは、コンテナを隔離する技術だけでなく、「イメージ」という共通フォーマットを通じて、世界中の開発者がソフトウェアを共有できるエコシステム（Docker Hub等）が存在することにあります。本章では、初心者が見落としがちな「イメージの実態」と「ベースOSの正しい選び方」について解説します。

## 2.1 Docker Registry と Docker Hub

私たちが `docker pull nginx` や `docker run ubuntu` と打った時、イメージはどこから来るのでしょうか？
デフォルトでは、Dockerは **Docker Hub** というパブリックな「イメージレジストリ（保管庫）」と通信します。

*   **Registry (レジストリ):** Dockerイメージを保存・配信するためのサーバー・アプリケーション。AWS ECR や GitHub Container Registry など、自社専用のプライベートレジストリを構築することも可能です。
*   **Repository (リポジトリ):** 同じアプリケーションのイメージを束ねたもの。例えば `nginx` リポジトリには、バージョンごとの様々なイメージが含まれています。
*   **Image (イメージ):** 実際にダウンロードされる実体。

### 自分のイメージを世界に公開する
自分で作ったイメージをDocker Hubにアップロード（Push）して共有する基本的なフローは以下の通りです。

```bash
# 1. Docker Hubにログイン
docker login

# 2. イメージに「自分のユーザー名/リポジトリ名:タグ」という形式で名前を付ける
docker tag my-custom-nginx:latest sen469/my-nginx:1.0

# 3. アップロード
docker push sen469/my-nginx:1.0
```

## 2.2 「latest」タグの罠とセマンティックバージョニング

イメージ名のコロン(`:`)以降の部分を **Tag（タグ）** と呼びます。
タグを指定せずに `docker pull nginx` と実行すると、Dockerは自動的に `nginx:latest` というタグを探します。

初学者の多くがこの `latest` を使い続けますが、**本番環境やチーム開発において `latest` タグの使用は厳禁（アンチパターン）です**（Inception課題でも明示的に禁止されています）。

*   **理由:** `latest` は「最新バージョン」を保証する魔法のタグではなく、単なる「最後にPushされたイメージに付与されるデフォルトの文字列」に過ぎません。ある日突然メジャーアップデートが行われ、昨日まで動いていたシステムが今日になって全く動かなくなる（破壊的変更）リスクがあります。

**ベストプラクティス:**
必ず `nginx:1.21.6` のように、バージョンが固定されたタグ（セマンティックバージョニング）を指定してください。これにより、何年経っても完全に同じ環境を再現できるようになります。

## 2.3 ベースイメージの選択: Debian vs Alpine vs Scratch

Dockerfileの1行目に書く `FROM <イメージ>`。ここで何を選ぶかが、イメージのサイズやセキュリティに直結します。

### 1. Debian / Ubuntu (汎用OSベース)
*   **特徴:** 普段使っているLinuxとほぼ同じ感覚で使えます。`apt-get` などのパッケージマネージャが標準で揃っており、ライブラリも豊富です。
*   **メリット:** トラブルシューティングが容易。コンパイル済みのバイナリがそのまま動くことが多い（glibcを使用）。
*   **デメリット:** サイズが大きく（数MB〜数百MB）、不要なツールが含まれているためセキュリティの攻撃面（Attack Surface）が広くなります。

### 2. Alpine Linux (軽量OSベース)
*   **特徴:** コンテナ向けに徹底的に最適化された超軽量Linuxディストリビューションです。
*   **メリット:** ベースイメージのサイズがわずか **5MB程度**。イメージのPullが非常に速く、不要なツールがないためセキュアです。
*   **デメリット:** Cライブラリとして `glibc` ではなく `musl libc` を使用しています。そのため、Pythonの特定のライブラリ（C言語で書かれた拡張モジュールなど）をインストールする際にコンパイルエラーになり、解決に高度な知識が必要になることがあります。

### 3. Scratch (空のイメージ)
*   **特徴:** 何も入っていない、完全に空のイメージです（サイズ 0 バイト）。
*   **用途:** Go言語やRust、C++などで「静的リンク」された（外部ライブラリに依存しない）単一の実行可能バイナリを動かすためだけに使われます。極限の軽量化とセキュリティを実現する、プロフェッショナルなアプローチです。

# 第3章: Dockerfile詳解とイメージビルドの最適化

コンテナの振る舞いを決定づけるのが「Dockerイメージ」です。Docker Hubから既存のイメージを引っ張ってくるだけでなく、独自のソフトウェアや設定を組み込んだカスタムイメージを作成するためには `Dockerfile` を記述する必要があります。本章では、単に動くイメージを作るだけでなく、軽量で安全、かつキャッシュ効率の良い「プロフェッショナルなイメージ」を作るための内部メカニズムを解説します。

## 3.1 Infrastructure as Code (IaC) としてのDockerfile

Dockerfileは、アプリケーションの実行環境（OS、ミドルウェア、ライブラリ、ソースコード、起動コマンド）をテキストベースの命令セットとして定義したものです。これにより、インフラストラクチャの構築が「属人的な手作業」から「Gitでバージョン管理可能なコード」へと昇華されました。

## 3.2 イメージレイヤーとキャッシュのメカニズム

Dockerイメージは1つの巨大なファイルではなく、複数の「読み取り専用レイヤー」が重なったミルフィーユのような構造（UnionFS）をしています。Dockerfileの命令（`FROM`, `RUN`, `COPY` など）を1行実行するたびに、新しいレイヤーが1つ作成されます。

Dockerビルドを高速化する最大の鍵は「**キャッシュ機構**」の理解です。
Dockerデーモンはビルド時、以前ビルドしたレイヤーのキャッシュが再利用できないか確認します。しかし、**「ある行でキャッシュが無効化されると、それ以降のすべての行でキャッシュが無効になる」**という絶対的なルールがあります。

**悪い例:**
```dockerfile
FROM debian:bullseye
COPY . /app/          # ソースコードをすべてコピー
RUN apt-get update    # パッケージリスト更新
RUN apt-get install -y nginx  # NGINXインストール
```
この書き方では、ソースコードを1文字でも修正すると `COPY` 行のキャッシュが無効になり、その後の `apt-get` が毎回ゼロから実行されてしまい、ビルドに膨大な時間がかかります。

**良い例 (キャッシュの最適化):**
```dockerfile
FROM debian:bullseye
RUN apt-get update && apt-get install -y nginx && rm -rf /var/lib/apt/lists/*
COPY . /app/
```
パッケージのインストールなど「変更されにくい処理」を上に、ソースコードのコピーなど「頻繁に変更される処理」を下に配置することで、キャッシュの恩恵を最大限に受けることができます。さらに `RUN` を `&&` で繋ぐことでレイヤー数を減らし、イメージサイズを小さく保つテクニックも重要です。

## 3.3 主要なインストラクションの深掘り

### COPY vs ADD
どちらもホストのファイルをコンテナ内に配置する命令ですが、明確な使い分けが必要です。
*   `COPY`: 単純なファイルやディレクトリのコピー。**通常はこちらを推奨します。**
*   `ADD`: `COPY` の機能に加え、URLからのダウンロードや、tarアーカイブの自動展開機能を持っています。予測不可能な挙動を避けるため、アーカイブ展開が目的でない限り `COPY` を使うのがベストプラクティスです。

### CMD vs ENTRYPOINT
コンテナ起動時に実行されるプロセスを指定する命令ですが、初心者が最もつまずきやすいポイントです。
*   `ENTRYPOINT`: コンテナを「特定の実行ファイルとして振る舞わせる」ための命令。基本的には上書きされません。
*   `CMD`: 起動時のデフォルトの引数やコマンドを提供します。`docker run` の引数によって簡単に上書きされます。

**Shell form と Exec form の違い:**
```dockerfile
# Shell form (非推奨)
CMD nginx -g "daemon off;"
# 実は背後で `sh -c 'nginx -g "daemon off;"'` として実行される。PID 1 が sh になってしまう。

# Exec form (推奨)
CMD ["nginx", "-g", "daemon off;"]
# 直接 nginx が実行される。NGINX自身が PID 1 を獲得する。
```

## 3.4 Inception課題における最重要要件: PID 1問題

Inceptionの課題要件には「コンテナをハック的な手法（`tail -f` や無限ループ）で起動してはならない」と明記されています。これはLinuxのプロセス管理と深く関わっています。

Linuxにおいて、起動した最初のプロセスは **PID 1** となります。PID 1 には以下の特別な責任が課せられます。
1.  子プロセスの終了状態を回収する（ゾンビプロセスの防止）。
2.  OS（またはDockerデーモン）から送られてくるシグナル（`SIGTERM`など）を適切にハンドリングし、自身と子プロセスを安全に終了させる。

もし `tail -f /dev/null` や無限ループスクリプトを `ENTRYPOINT` にしてコンテナを起動すると、それらが PID 1 になります。これらはシグナルを適切に処理する機能を持たないため、`docker stop` を実行してもコンテナがスムーズに終了せず、強制終了（`SIGKILL`）されるまで10秒間フリーズしてしまいます。さらにはゾンビプロセスが蓄積し、システムリソースを枯渇させる原因になります。

**正しい設計:**
コンテナの主役となるサービス（NGINXのマスタープロセスや、PHP-FPMのマスタープロセス、MariaDBデーモン）が直接 PID 1 を獲得し、フォアグラウンドで走り続けるように `ENTRYPOINT` または `CMD` を設計（Exec formを使用）しなければなりません。これがコンテナ設計の最も美しい「あるべき姿」です。

## 3.5 まとめ
本章では、Dockerfileを用いたイメージ構築と、IaCとしてのベストプラクティスを学びました。次章では、このビルドプロセスを本番環境向けに極限まで最適化する「マルチステージビルド」について解説します。
# 第4章: マルチステージビルドと本番環境への最適化

Dockerを学び始めたばかりの頃は、「動けばいい」という理由で `FROM ubuntu` のような巨大なベースイメージを選び、そこに開発ツールを詰め込んだまま本番環境にデプロイしがちです。しかし、プロフェッショナルな現場では、イメージの「サイズ」と「セキュリティ」を極限まで追求する必要があります。その最適解となる機能が **マルチステージビルド (Multi-stage Builds)** です。

## 4.1 開発環境と本番環境の矛盾

Go言語、Rust、C++、あるいはNode.js（TypeScript）などのアプリケーションをビルドする際、以下のような矛盾が生じます。

1.  **ビルド時に必要なもの:** コンパイラ（gcc, go, tsc等）、ビルドツール（make, cmake）、ソースコードの全体。
2.  **実行時（本番）に必要なもの:** コンパイルされた**実行可能バイナリファイル**（またはトランスパイルされたJS）と、最小限のランタイムだけ。

以前は、これらを分離するために「ビルド用コンテナでコンパイルし、出力されたバイナリをホストOSに取り出し、別の軽量コンテナに `COPY` する」という煩雑なシェルスクリプトを書いていました。これを一つの `Dockerfile` の中で完結させる魔法の機能が、マルチステージビルドです。

## 4.2 マルチステージビルドの書き方

以下は、Go言語のアプリケーションをマルチステージビルドで構築する典型的な例です。

```dockerfile
# ----------------------------------------
# ステージ 1: ビルド環境 (Builder)
# ----------------------------------------
FROM golang:1.21-alpine AS builder

WORKDIR /app
# 依存関係のダウンロードとソースコードのコピー
COPY go.mod go.sum ./
RUN go mod download
COPY . .

# アプリケーションのコンパイル（CGO無効化で静的バイナリを作成）
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o myapp .

# ----------------------------------------
# ステージ 2: 実行環境 (Production)
# ----------------------------------------
FROM scratch

# ビルドステージ（builder）からコンパイル済みのバイナリ「のみ」をコピーする
COPY --from=builder /app/myapp /myapp

# ポートの公開と実行
EXPOSE 8080
ENTRYPOINT ["/myapp"]
```

### 何が起きているのか？
1.  **`AS builder`**: 最初の `FROM` に名前（エイリアス）を付けます。このステージは、コンパイラなどの重いツールを含む「使い捨ての作業場」です。
2.  **2つ目の `FROM scratch`**: 新しいステージを開始します。この瞬間、**これまでのレイヤー（ファイル群）はすべて破棄され、真っ新な状態からスタートします。**
3.  **`COPY --from=builder`**: 唯一の繋がりとして、前のステージから特定のファイル（コンパイル済みのバイナリ）だけを抜き出してコピーします。

結果として生成される最終的な Docker イメージには、ソースコードも、Goコンパイラも、OSの基本コマンド群すら含まれていません。イメージサイズは **数MB** にまで劇的に縮小され、ハッカーが侵入してシェル（`sh`）を起動しようとしても、シェル自体が存在しないため何もできないという究極のセキュリティ（Distrolessアプローチ）が実現します。

## 4.3 スクリプト言語（Node.js, Python）での活用

コンパイル言語だけでなく、Node.jsやPythonのようなスクリプト言語でもマルチステージビルドは有効です。

*   **Node.jsの例:** `builder` ステージで TypeScript のトランスパイルと `npm install`（開発用パッケージ含む）を実行し、本番用ステージにはコンパイルされた `.js` ファイルと `node_modules`（本番用パッケージのみ）だけをコピーする。
*   **Pythonの例:** `builder` ステージで `gcc` などのビルドツールを使ってC言語依存のライブラリをWheel形式でビルドし、本番用ステージには軽量なAlpineをベースにしてWheelだけをインストールする。

## 4.4 まとめ
マルチステージビルドは、不要なゴミを一切含まない純粋な「本番用コンテナ」を構築するための現代のデファクトスタンダードです。次章からは、単一のコンテナを離れ、複数のコンテナを統合的にオーケストレーションする「Docker Compose」のアーキテクチャへと進みます。
# 第5章: Docker Composeによるマルチコンテナ・オーケストレーション

前章までで、単一のコンテナを制御する方法と、その元となるイメージの設計図（Dockerfile）の書き方を学びました。しかし、実際のシステム構築（Inception課題を含む）では、Webサーバー、アプリケーション、データベースなど複数のコンテナが複雑に連携して動作します。これらを `docker run` コマンドだけで管理するのは非現実的です。そこで登場するのが **Docker Compose** です。

## 5.1 宣言的インフラストラクチャへのパラダイムシフト

`docker run` は「これを実行しろ」と命令する **命令的（Imperative）** なアプローチです。
対して `docker-compose.yml` は「システムはこうあるべきだ」という最終的な状態を定義する **宣言的（Declarative）** なアプローチを採用しています。

Composeツールは、現在のシステム状態とYAMLファイルに書かれた「あるべき状態」の差分を計算し、必要なコンテナだけを作成・再起動してくれます。これにより、複雑なシステム全体を `docker-compose up -d` というたった一つのコマンドで、誰の環境でも確実に再現できるようになりました。

## 5.2 `docker-compose.yml` の構造解剖

Composeファイルは主に3つのトップレベル要素（`services`, `networks`, `volumes`）から構成されます。

### 1. Services (コンテナの定義)
各コンテナの設定を記述します。
```yaml
services:
  nginx:
    build:
      context: ./requirements/nginx
      dockerfile: Dockerfile
    image: inception_nginx
    ports:
      - "443:443"
    depends_on:
      - wordpress
    networks:
      - inception_network
```
*   **build**: Inceptionの課題では既存イメージ（`image: nginx`）の直接使用が禁止されています。代わりに `build` ディレクティブを使って、指定したディレクトリ（`context`）にある `Dockerfile` を基にローカルでイメージを構築させます。
*   **depends_on**: コンテナの起動順序を制御します。上記の場合、`wordpress` が起動した*後*に `nginx` が起動します（ただし、アプリケーションの準備完了までは待機しない点に注意が必要です）。

### 2. Networks (内部ネットワークとサービスディスカバリ)
Docker Composeにおける最も強力な機能の一つが、コンテナ間の名前解決（DNS）です。
```yaml
networks:
  inception_network:
    driver: bridge
```
同じカスタムネットワーク（`inception_network`）に所属するコンテナ同士は、**サービス名（例: `mariadb`, `wordpress`）をそのままホスト名として通信できます。**
Dockerエンジンに内蔵されたDNSサーバーが、サービス名を自動的にコンテナの内部IPアドレス（例: `172.18.0.3`）に変換（サービスディスカバリ）してくれます。したがって、動的に変わるIPアドレスをハードコードする必要は一切ありません。

*※Inceptionの課題では `network: host`（ホストのネットワークを共有する設定）や、古い非推奨機能である `--link` の使用は明示的に禁止されています。必ずカスタムネットワークを定義してください。*

### 3. Volumes (永続化データ領域)
コンテナは「使い捨て」を前提として設計されているため、コンテナを削除すると内部のデータも消滅します。これを防ぐためにボリュームを定義します（詳細なメカニズムは次章で解説します）。

```yaml
volumes:
  wordpress_data:
    driver: local
```

## 5.3 環境変数と構成の分離

インフラのコード化において、「コード（構成）」と「環境変数（パスワードやドメイン名）」は分離すべきです。Composeでは、同一ディレクトリにある `.env` ファイルを自動的に読み込みます。

```yaml
services:
  mariadb:
    # ...
    environment:
      MYSQL_DATABASE: ${MYSQL_DATABASE_NAME}
      MYSQL_USER: ${MYSQL_USER}
```
このように記述することで、認証情報がGitHubなどのリポジトリに漏洩するリスクを防ぎ、本番環境と開発環境で異なる設定を簡単に注入することができます。

## 5.4 まとめ
Docker Composeを活用することで、システム全体を単一のコードベースとして統合的に管理できるようになります。次章では、複数コンテナの運用において最も慎重な設計が求められる「ネットワーク」と「データの永続化（Volumes）」について深く掘り下げます。
# 第6章: ネットワークアーキテクチャとデータの永続化

コンテナ自体は揮発性（使い捨て）であり、独立した隔離空間で動作します。しかし、実用的なシステムを構築するためには、「コンテナ間の通信（ネットワーク）」と「コンテナ破棄後も残すべきデータ（ボリューム）」を適切に設計・管理しなければなりません。本章では、Dockerにおけるネットワークドライバとストレージマウントの仕組みを深く理解します。

## 6.1 Dockerネットワークドライバの仕組み

Dockerは、用途に応じて複数のネットワークドライバを提供しています。

1.  **bridge (デフォルト):** 単一のホストマシン内でコンテナ同士を接続するためのプライベートな内部ネットワーク。外部からはNAT（Network Address Translation）経由でアクセスします。
2.  **host:** コンテナのネットワーク隔離を無効化し、ホストマシンのネットワークインターフェースを直接共有します。ポートの競合が起きやすいため、セキュリティ要件の厳しいInception課題では**使用が固く禁じられています**。
3.  **overlay:** 複数のDockerホスト（Swarmクラスタなど）にまたがるコンテナ群を接続するための高度なネットワーク。
4.  **macvlan:** コンテナに物理的なMACアドレスを割り当て、ネットワーク上の独立した物理デバイスのように見せかける技術。

### デフォルトブリッジ vs カスタムブリッジ
`docker run` でネットワークを指定しなかった場合、デフォルトの `bridge` ネットワーク（`docker0`）に接続されます。しかし、このデフォルトブリッジには**「コンテナ名による自動名前解決（DNSサービスディスカバリ）が機能しない」**という致命的な弱点があります。
一方、`docker network create` や Docker Compose で定義した**カスタムブリッジ**では、内蔵DNSが有効になり、IPアドレスを意識することなく `ping mariadb` のような通信が可能になります。これがカスタムネットワークを定義すべき最大の技術的理由です。

## 6.2 データの永続化戦略とマウント方式

コンテナ内のファイルシステム（OverlayFS）に書き込まれたデータは、コンテナを `docker rm` で削除すると全て消滅します。データベースのレコードやアップロードされた画像ファイルを保護するために、ホストマシンのファイルシステムをコンテナ内に「マウント（接合）」する必要があります。

Dockerには主に3つのマウント方式が存在します。

### 1. Bind Mounts (バインドマウント)
ホストOS上の「任意の絶対パス（例: `/home/user/my_project/src`）」をコンテナ内に直接マウントします。
*   **利点:** ホスト側でソースコードを編集すると即座にコンテナ内に反映されるため、ローカル開発環境で非常に便利。
*   **欠点:** ホストのディレクトリ構造に強く依存するため、環境間のポータビリティ（移植性）が低下する。また、ホスト側の権限設定（UID/GID）によるパーミッションエラーが起きやすい。

### 2. Named Volumes (名前付きボリューム)
Dockerデーモン自身が管理する専用の領域（通常は `/var/lib/docker/volumes/` 配下）にデータを保存します。
*   **利点:** Dockerがライフサイクルを完全に管理するため、バックアップや移行が容易。ホストOSのファイル構造に依存せず、プラットフォーム間での互換性が極めて高い。
*   **Inceptionでの必須要件:** 課題では、データベースとWordPressファイルの永続化にこの「Named Volumes」を使用することが**厳格に義務付けられています**（Bind Mountsの使用は禁止）。

### 3. tmpfs Mounts
ホストのディスク（HDD/SSD）ではなく、システムのメモリ（RAM）上に一時的な領域を確保してマウントします。高速ですが揮発性であり、シークレットキーなどの機密情報を安全に扱う際に用いられます。

## 6.3 Inception特有の要件: Named Volumeの高度な制御

Inceptionの課題には非常にトリッキーな要件が存在します。
「**Named Volumes を使わなければならないが、そのデータの実際の保存先はホストマシンの `/home/login/data` 配下でなければならない**」というものです。

これは一見矛盾しています。通常の Named Volume は保存先をDockerに一任する（`/var/lib/docker/volumes/...`）からです。この要件を満たすためには、Docker Composeの `volumes` 定義において `driver_opts` を駆使し、「Dockerが管理するNamed Volumeの実体を、ホストの特定パスにバインドさせる」という高度な設定を行う必要があります。

**実装例 (`docker-compose.yml`):**
```yaml
volumes:
  wordpress_data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/${USER}/data/wordpress  # ホスト側の実体パス
```
*※注: `device` で指定したパスのディレクトリは、コンテナ起動前にホスト側で作成されている（存在している）必要があります。存在しない場合、Dockerはマウントエラーを返します。*

## 6.4 まとめ
本章では、コンテナを単なる計算リソース（CPU/メモリ）から、ステートフルなシステムへと進化させるためのネットワークとストレージの技術を学びました。次章では、ここで永続化されたボリュームをどのように安全にバックアップし、別環境へ移行するかという「運用（オペレーション）」のベストプラクティスを解説します。
# 第7章: データのバックアップ・復元と移行戦略

Dockerの「データ永続化（Named Volume）」については前章で学びましたが、現場で必ず直面するのが「**その保存されたデータをどうやってバックアップし、別のサーバーに移行するのか？**」という運用（オペレーション）の課題です。本章では、コンテナ時代のデータ管理手法について実践的なアプローチを解説します。

## 7.1 ホストOSからの直接アクセス（アンチパターン）

Named Volumeのデータは、ホストマシンの `/var/lib/docker/volumes/[ボリューム名]/_data` に物理的に存在しています。
「だったら、このフォルダをそのまま `cp` や `tar` でコピーすればいいのでは？」と考えるのは自然ですが、これは**非常に危険なアンチパターン**です。

**理由:**
1.  **権限問題:** コンテナ内のプロセス（例: `mysql` ユーザー）が作成したファイルは、ホスト側からは root 権限でしかアクセスできないことが多く、パーミッションが複雑に絡み合っています。
2.  **不整合（データコラプション）:** データベース（MariaDBなど）が稼働中でメモリ上のデータをディスクに書き込んでいる最中に直接コピーすると、ファイルが破損して復元不可能になる可能性が高いです。
3.  **Docker Desktopの制約:** MacやWindowsのDocker Desktop環境では、このパスはホスト（Mac/Win）上に存在するのではなく、Dockerが隠蔽している「軽量なLinux仮想マシン」の中に存在するため、そもそも直接アクセスできません。

## 7.2 一時コンテナによるバックアップ手法（ベストプラクティス）

安全かつ環境に依存しない標準的なバックアップ手法は、「**バックアップ専用の使い捨てコンテナを一時的に起動し、そのコンテナ経由でホストとデータをやり取りする**」というアプローチです。

### データのバックアップ手順

以下は、`db_data` というボリュームの中身を `backup.tar` というファイルに固めて、ホストのカレントディレクトリに取り出すコマンドです。

```bash
docker run --rm \
  -v db_data:/source:ro \
  -v $(pwd):/backup \
  alpine \
  tar cvf /backup/backup.tar -C /source .
```

**コマンドの解剖:**
*   `--rm`: 処理が終わったらこの一時コンテナを自動削除する。
*   `-v db_data:/source:ro`: バックアップ対象のボリュームを、コンテナ内の `/source` に**読み取り専用(ro)**でマウントする。
*   `-v $(pwd):/backup`: ホストマシンのカレントディレクトリを、コンテナ内の `/backup` にマウントする。
*   `alpine`: 軽量なAlpineイメージを使用する。
*   `tar cvf /backup/backup.tar -C /source .`: コンテナ内で `tar` コマンドを実行し、`/source` の中身をまとめて `/backup` (ホストと繋がっている) に保存する。

### データの復元（リストア）手順

逆に、新しい環境（または初期化した環境）で空のボリューム `new_db_data` を作成し、そこに先ほどの `backup.tar` を展開してデータを復元する手順です。

```bash
# まず空のボリュームを作成
docker volume create new_db_data

# バックアップをボリュームに展開
docker run --rm \
  -v new_db_data:/target \
  -v $(pwd):/backup:ro \
  alpine \
  tar xvf /backup/backup.tar -C /target
```

## 7.3 データベースの論理バックアップ（ダンプ）

ファイルレベルのバックアップ（上記）に加え、MariaDBやPostgreSQLなどのデータベースでは「論理バックアップ」も併用すべきです。これはデータベース特有のコマンド（`mysqldump`など）を使って、データをSQL文として書き出す方法です。

```bash
# 稼働中のMariaDBコンテナからデータをダンプしてホストに保存
docker exec mariadb sh -c 'exec mysqldump --all-databases -u root -p"$MYSQL_ROOT_PASSWORD"' > db_dump.sql
```
この手法であれば、データベースが稼働中でもデータの不整合を気にすることなく、安全にバックアップを取得できます。

## 8.4 まとめ
コンテナは使い捨てですが、データは企業の生命線です。「コンテナの外（ホスト）から触る」のではなく、「コンテナの世界のルール（マウントと一時コンテナ）に従ってデータを出し入れする」という感覚を身につけることが、Docker運用スキルの第一歩となります。次章では、システムを外部に公開する上で最も重要な「セキュリティの要諦と機密情報の管理」について解説します。
# 第8章: セキュリティの要諦と機密情報の管理

システムを外部（インターネット）に公開する際、セキュリティは後から追加するオプションではなく、設計段階から組み込まれるべき中核要素（Security by Design）です。本章では、Dockerコンテナ環境におけるクレデンシャル（パスワードやAPIキー）の安全な取り扱いと、ネットワーク通信の暗号化について解説します。

## 8.1 機密情報のハードコードとその危険性

**「Dockerfileやソースコードにパスワードを直接書いてはいけない」**
これは開発における絶対的なルールです。コードに直接書き込まれた（ハードコードされた）機密情報は、Gitでコミットされた瞬間にリポジトリの履歴に永久に残り、世界中の誰でも（たとえプライベートリポジトリであっても、アクセス権を持つ全員が）閲覧可能になります。これを防ぐために、機密情報は「環境」から注入する必要があります。

## 8.2 環境変数（Environment Variables）とその限界

Dockerにおいて最も一般的な構成値の注入方法は「環境変数」です。Docker Composeでは `.env` ファイルを使用します。

**`.env` ファイルの役割:**
`.env` ファイルに `MYSQL_ROOT_PASSWORD=my_secure_password` と記述し、このファイル自体を `.gitignore` に追加してGitの管理から除外します。これにより、コードベースを汚染することなくコンテナにパスワードを渡すことができます。

**環境変数のセキュリティ上の限界:**
`.env` ファイルはハードコードよりはるかに安全ですが、完璧ではありません。環境変数として渡された値は、以下の方法で比較的容易に覗き見ることができてしまいます。
1.  **`docker inspect <コンテナ名>`**: このコマンドを実行すると、コンテナの全設定がJSONで出力され、その中の `Env` 配列に平文のパスワードが表示されます。
2.  **プロセスの環境変数**: コンテナに侵入された場合、`printenv` コマンドを打つだけで全環境変数が漏洩します。

## 8.3 Docker Secretsのアーキテクチャ（推奨要件）

より高度な機密情報管理を行うために提供されているのが **Docker Secrets** です。Inceptionの課題でも「Secretsの使用が強く推奨される」と明記されています。

**Docker Secretsの仕組み:**
1.  ホスト側のファイル（例: `secrets/db_password.txt`）にパスワードを保存します。
2.  Composeファイルで `secrets` を定義し、コンテナにマウントします。
3.  Dockerは、コンテナ内の特定のディレクトリ（通常は `/run/secrets/`）に、そのパスワードファイルを**メモリ上の揮発性ファイルシステム（tmpfs）としてマウント**します。

**なぜ安全なのか？**
*   **環境変数に露出しない:** `docker inspect` や `printenv` を実行しても、パスワードは見えません。
*   **ディスクに書き込まれない:** tmpfsとしてマウントされるため、コンテナのディスク上にパスワードが物理的に保存されることはありません。
*   アプリケーション（例: WordPressやMariaDBの起動スクリプト）は、`/run/secrets/db_password` という「ファイル」を読み取ってパスワードを取得します。

## 8.4 ネットワーク境界防御（NGINXとTLS）

システムの入り口（エントリポイント）を一つに絞り、そこで通信を暗号化・検証することは、ネットワークセキュリティの基本です。Inceptionアーキテクチャでは、**NGINXコンテナのみがポート443（HTTPS）を外部に公開し、他のコンテナ（WordPress, MariaDB）は内部ネットワークに完全に隠蔽されます。**

### TLSプロトコルの制限 (TLS 1.2 / 1.3)
HTTPS通信の基盤となるTLS（Transport Layer Security）プロトコルは、時代とともに進化してきました。SSLv3 や TLS 1.0 / 1.1 には、POODLEやBEASTといった既知の深刻な脆弱性が存在します。

課題の要件にある「TLSv1.2 または TLSv1.3 のみを使用する」という制限は、NGINXの設定ファイルにおいて以下のように明示的に指定することで満たされます。

```nginx
server {
    listen 443 ssl;
    listen [::]:443 ssl;

    server_name ${DOMAIN_NAME};

    ssl_certificate     /etc/nginx/ssl/inception.crt;
    ssl_certificate_key /etc/nginx/ssl/inception.key;

    # ここが最重要：古い脆弱なプロトコルを無効化する
    ssl_protocols TLSv1.2 TLSv1.3;
}
```

## 8.5 まとめ
本章では、機密情報をコードや環境変数から分離し、メモリ上で安全に扱う手法（Secrets）と、単一のエントリポイントで強力な暗号化（TLS 1.2/1.3）を強制する境界防御の概念を学びました。次章では、これまでの全8章で培った知識を総動員し、三層アーキテクチャ（Web・App・DB）を構築する実践的なケーススタディ（Inceptionスタック）の設計を行います。
# 第9章: Inceptionスタックの統合とアーキテクチャ設計

これまでの8つの章で、Dockerの基盤技術からネットワーク、ストレージ、運用管理、セキュリティに至るまでのコンポーネントを学んできました。本章はそれらの集大成として、Inception課題で求められている「NGINX - WordPress - MariaDB」の三層アーキテクチャをどのように統合し、連携させるかを解説します。

## 9.1 三層アーキテクチャ (3-Tier Architecture) の全体像

Inceptionスタックは、現代のWebアプリケーションの標準的な設計である三層アーキテクチャを採用しています。

1.  **プレゼンテーション層 (Web Server):** NGINXコンテナ。ユーザーからのHTTPSリクエストを受け付け、静的ファイル（画像やCSS）を返し、動的処理を後段に転送します。
2.  **アプリケーション層 (App Server):** WordPress + PHP-FPMコンテナ。ビジネスロジック（PHPコード）を実行し、HTMLを動的に生成します。
3.  **データ層 (Database):** MariaDBコンテナ。ユーザー情報や投稿記事などの永続的なデータを保存します。

この三層をそれぞれ独立したコンテナに分離することで、特定の層だけをスケールさせたり、個別にアップデートしたりすることが容易になります（疎結合の実現）。

## 9.2 NGINX: リバースプロキシとFastCGI

Inception要件の大きな罠の一つが、「WordPressコンテナにNGINXを含めてはいけない（NGINXなしでWordPressを動かす）」という制約です。

通常、PHPアプリケーションを動かす際はApacheなどのWebサーバーモジュールとして組み込まれることが多いですが、今回は **PHP-FPM (FastCGI Process Manager)** というスタンドアロンのプロセスを使用します。

### データフローのメカニズム
1.  ユーザーがブラウザで `https://login.42.fr/index.php` にアクセスします。
2.  ポート443で待ち受けている **NGINX** がリクエストを復号化（TLS終端）します。
3.  NGINXはリクエストが `.php` ファイルであることを検知し、自身の `fastcgi_pass` 設定に従って、**Docker内部ネットワーク経由で WordPressコンテナのポート9000** へ転送します。
4.  WordPressコンテナ内で稼働している **PHP-FPM** プロセスがリクエストを受け取り、`index.php` を実行します。

**NGINXのコンフィグ例 (概念):**
```nginx
location ~ \.php$ {
    # wordpressコンテナの9000番ポートへ転送（サービス名で名前解決）
    fastcgi_pass wordpress:9000;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}
```
ここで重要なのは、NGINXコンテナとWordPressコンテナの両方が、**WordPressのソースコード（`.php` ファイル群）へのアクセス権を持っていなければならない**という点です。したがって、WordPressファイルを格納する「Named Volume」は、これら両方のコンテナにマウントされる必要があります。

## 9.3 WordPress + PHP-FPM: 自動化と実行層

WordPressコンテナの主役は PHP-FPM プロセス（これがPID 1になります）です。
しかし単にPHP-FPMを起動するだけでは、WordPressの初期設定（データベース接続情報の入力、管理者ユーザーの作成など）を手動でブラウザから行わなければなりません。これは「IaCの完全自動化」の理念に反します。

これを解決するために、コンテナ起動時（`ENTRYPOINT` スクリプト内）で **WP-CLI (WordPress Command Line Interface)** を使用します。
WP-CLIを使えば、データベースのセットアップ、サイトURLの構成、管理者（admin以外）と一般ユーザーの作成といったすべての初期化プロセスを、コマンドラインから自動で完了させることができます。

## 9.4 MariaDB: データ層の隔離と保護

データ層であるMariaDBコンテナは、他の層から完全に隔離されていなければなりません。

*   **ポートの非公開:** `docker-compose.yml` において、`ports: - "3306:3306"` といったホスト側へのポートマッピングは**絶対に記述してはいけません**。データベースは内部ネットワーク経由でWordPressコンテナからのみアクセス可能であるべきです。
*   **初期化の自動化:** MariaDBイメージでは、起動時のエントリポイントスクリプトで `.sql` スクリプトを流し込むか、特定の環境変数（またはSecrets）を読み込んで、自動的に「WordPress用データベース」と「専用ユーザー」を作成し、権限（GRANT）を付与する仕組みを構築する必要があります。

## 9.5 トラブルシューティングの勘所

この複雑なスタックを構築する際、必ずエラーに直面します。その際のアプローチを整理しておきましょう。

1.  **502 Bad Gateway:** NGINXがWordPressコンテナの9000番ポートに接続できていません。PHP-FPMが落ちているか、ネットワーク設定（サービス名）が間違っています。
2.  **データベース接続エラー:** WordPressがMariaDBに接続できていません。データベースの起動完了前にWordPressが接続を試みているか、ユーザー名/パスワード（Secrets）の受け渡しに失敗しています。
3.  **File Not Found (NGINX):** NGINXコンテナ側に、WordPressの静的ファイルが存在するボリュームが正しくマウントされていません。

## 9.6 まとめ
三層アーキテクチャの構築は、単なる環境構築を超えた「システム設計の美学」を学ぶプロセスです。各コンテナが単一の責任（Single Responsibility）を持ち、セキュアな内部ネットワークで連携し、外部からの攻撃面を最小限に抑える。この設計思想を理解し、自身の言葉で README や DEV_DOC に説明できるようになることが、本プロジェクトの最終目標です。
# 第10章: 付録 - プロフェッショナルのためのDockerコマンド・リファレンス

インフラ構築の現場において、コマンドは単なる暗記対象ではなく「システムと対話するための語彙」です。本章では、Inception課題や実際の業務で頻繁に使用するコマンド群を、その背後にある挙動の解説とともに体系的にまとめました。

## 7.1 Compose ライフサイクル管理

プロジェクト全体（複数コンテナ）のオーケストレーションを行うコマンドです。基本的には `docker-compose.yml` が存在するディレクトリで実行します。

| コマンド | 背後の挙動と解説 |
| :--- | :--- |
| `docker-compose up -d --build` | イメージを再ビルドし、コンテナ群を作成してバックグラウンド(`-d`)で起動します。変更を確実に反映させるための最も標準的な起動コマンドです。 |
| `docker-compose down` | コンテナを停止し、さらにコンテナとデフォルトネットワークを**破棄**します。ただし、データボリュームは保持されます。 |
| `docker-compose down -v` | コンテナとネットワークに加え、**定義されたデータボリュームも完全に破棄**します。データベースを初期状態に戻したい（ゼロからやり直したい）場合に必須です。 |
| `docker-compose restart [サービス名]` | 指定したサービス（例: `nginx`）のプロセスに `SIGTERM` を送り、再起動させます。設定ファイルの変更を反映させたい時に使います。 |

## 7.2 インスペクションとデバッグ

コンテナが期待通りに動かない（あるいはすぐに落ちてしまう）場合の、原因究明のためのコマンド群です。

| コマンド | 背後の挙動と解説 |
| :--- | :--- |
| `docker-compose logs -f [サービス名]` | サービスの標準出力・標準エラー出力をリアルタイム(`-f`: tail -fと同等)で監視します。NGINXの502エラーやMariaDBの初期化失敗の原因はほぼすべてここに記載されます。 |
| `docker exec -it [コンテナ名] sh` | 稼働中のコンテナのNamespace内に新しい `sh` (または `bash`) プロセスを割り込ませ、対話型シェルを開きます。設定ファイルが正しくコピーされているか、コンテナ内から `ping mariadb` が通るか等の確認に使います。 |
| `docker inspect [コンテナ名]` | コンテナのすべてのメタデータ（JSON形式）を出力します。マウントされているボリュームの正確なホスト側パスや、注入された環境変数、IPアドレスを確認する最強のデバッグツールです。 |
| `docker network inspect [ネットワーク名]` | 指定したネットワークにどのコンテナが所属し、どのIPアドレスが割り当てられているかを確認します。名前解決エラーのトラブルシューティングに役立ちます。 |

## 7.3 リソース管理と徹底的なクリーンアップ

開発を繰り返していると、使われなくなった古いイメージや中途半端なボリュームが蓄積し、ホストOS（VM）のディスク容量を圧迫します。

| コマンド | 背後の挙動と解説 |
| :--- | :--- |
| `docker stats` | すべての稼働中コンテナのCPU使用率、メモリ消費量、ネットワークI/Oをリアルタイムで表示します（`top`コマンドのコンテナ版）。 |
| `docker system prune -a --volumes` | **[取扱注意]** 停止中のすべてのコンテナ、使われていないネットワーク、タグ付けされていない（または未使用の）すべてのイメージ、さらに**マウントされていないすべてのボリューム**をディスク上から物理的に消去します。プロジェクトを真っ更な状態からビルドし直して「評価要件を満たすか」を確認する最終テストの際に非常に強力です。 |
| `docker builder prune` | Dockerのビルドキャッシュをクリアします。「Dockerfileを書き換えたのに、なぜか古いキャッシュが使われて挙動が変わらない」という稀なケースに遭遇した場合に使用します。 |

## 8.4 評価（Defense）に向けてのワンポイント

評価の際、単に「動いていること」を示すだけでなく、**「なぜそのコマンドを打ったのか」「そのコマンドによってシステムに何が起きたのか」**を説明できることが求められます。
例えば、`docker-compose down -v` を打つ際は、「これでコンテナレイヤーだけでなく、永続化されていたボリューム（`/var/lib/docker/volumes/` にマウントされていたデータ）も削除され、次回の起動時にはMariaDBの初期化スクリプトが再び走るようになります」と説明できれば、技術への深い理解を示すことができます。
