# サービス解説 04: エントリポイントと PID 1 問題の完全解決

Dockerコンテナの設計において、最も重要かつ最もエラーが起きやすいのが **「コンテナが起動した瞬間に実行されるスクリプト（Entrypoint）」** の設計です。Inceptionの課題要件でも、プロセス管理（PID 1）について強く言及されています。

本章では、シェルスクリプトを使ってコンテナを正しく初期化し、安全にプロセスを起動する魔法のコマンド **`exec "$@"`** について解説します。

## 1. なぜ起動スクリプト（setup.sh）が必要なのか？

これまで見てきたように、WordPressやMariaDBのコンテナでは、単にソフトを起動するだけでなく「データベースの作成」や「WP-CLIによる初期設定」を自動で行う必要があります。
これを `Dockerfile` の `CMD` だけで表現するのは不可能なため、シェルスクリプト（`.sh`）を用意し、それを `ENTRYPOINT` に指定します。

**Dockerfileの例:**
```dockerfile
COPY setup.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/setup.sh
ENTRYPOINT ["/usr/local/bin/setup.sh"]
CMD ["php-fpm8.4", "-F"]
```

## 2. 致命的なバグ：シェルスクリプトが PID 1 になってしまう

さて、以下のような `setup.sh` を書いたとします。

**悪い setup.sh の例:**
```bash
#!/bin/bash

# 1. 色々な初期設定処理...
echo "Setup is running..."

# 2. 最後にメインのプログラム（例: php-fpm）を起動する
php-fpm8.4 -F
```

これがなぜ「悪い」のでしょうか。
コンテナが起動すると、Dockerは `ENTRYPOINT` に指定された `setup.sh` を **PID 1**（最初のプロセス）として実行します。
その後、スクリプトの一番下にある `php-fpm8.4 -F` が実行されますが、これはシェルスクリプトから呼び出された「子プロセス」として起動するため、PID は 2 や 10 になります。

この状態で `docker stop`（コンテナを止めろという指示）を実行するとどうなるでしょうか。
Dockerは PID 1 である `setup.sh` に対して「終了しろ」というシグナル（SIGTERM）を送ります。しかし、**単なるシェルスクリプトはシグナルを受け取って子プロセス（PHP-FPM）を安全に終了させる機能を持っていません。**

結果として、Dockerは「言うことを聞かない！」と判断し、10秒間フリーズした後に強制終了（SIGKILL）させてしまいます。これはデータ破損の原因になり、Inception課題でも不合格（Fail）になります。

## 3. 魔法のコマンド `exec "$@"` のOSレベルの仕組み (fork と execve)

この問題を解決するのが、Linuxの **`exec`** コマンドです。
`exec` は、「現在のプロセス（自分自身）を破棄し、指定されたプログラムに完全に中身をすり替える」という特殊なコマンドです。

これはLinuxカーネルの根本的なプロセス生成モデルに関わっています。Linuxにおいて、新しいプロセスを生み出す方法は **`fork()`** と **`execve()`** という2つのシステムコールの組み合わせしかありません。

1.  **通常の場合（`exec` を使わない場合）**
    シェルスクリプト内で `php-fpm8.4` と打つと、シェルはまず自分自身のコピー（クローン）を作ります（**`fork`**）。次に、そのクローンの中身を PHP-FPM のプログラムに入れ替えます（**`execve`**）。
    結果として、親プロセス（シェル: PID 1）と、子プロセス（PHP-FPM: PID 10）という2つのプロセスが存在することになります。
2.  **`exec` を使った場合**
    シェルスクリプト内で `exec php-fpm8.4` と打つと、シェルは **クローン（fork）を作るのをサボります**。
    そして、自分自身（PID 1のシェル）の脳みそ（メモリ空間）に、直接 PHP-FPM のプログラムを上書きロードしてしまいます（いきなり **`execve`**）。

**良い setup.sh の例:**
```bash
#!/bin/bash

# 1. 色々な初期設定処理...
echo "Setup is running..."

# 2. 魔法のコマンド！
exec "$@"
```

### `exec "$@"` の仕組み

1.  コンテナが起動し、`setup.sh` が **PID 1** として走り出します。初期設定処理が行われます。
2.  スクリプトが一番下の `exec "$@"` に到達します。
    *   `$@` は、「Dockerfileの `CMD` で渡された引数」を意味します。
    *   今回の例では `CMD ["php-fpm8.4", "-F"]` なので、`$@` は `php-fpm8.4 -F` に展開されます。
3.  つまり、`exec php-fpm8.4 -F` が実行されます。
4.  この瞬間、PID 1 として走っていた **シェルスクリプトは消滅し、その PID 1 という「座席」にそのまま PHP-FPM が座り直します（プロセスの置換）。**

これにより、最終的にコンテナの中で PID 1 として走り続けるのは PHP-FPM（または NGINX, MariaDB）自身になります。
彼らはシグナルを正しく受け取ることができるため、`docker stop` を打てば1秒で安全に綺麗に終了する（Graceful Shutdown）完璧なコンテナが完成します。

## 4. まとめ
コンテナの初期化スクリプトを書く際は、以下のテンプレートを必ず守りましょう。

1.  一番上には `#!/bin/bash` や `#!/bin/sh` を書く。
2.  真ん中に、自分がやりたい設定処理を書く。
3.  **一番最後は必ず `exec "$@"` で終わる。**
4.  Dockerfile では `ENTRYPOINT ["/setup.sh"]` と書き、`CMD` に本当に起動したいメインプログラムのコマンドを書く。

これが、プロフェッショナルなコンテナ設計の鉄則です。
